Using a Website Builder?

My aim was to create an app - not a website. I had not created a website in 7 years, and was rusty in coding and administration. I was a relatively good website full stack developer for someone who was self taught, but a poor administrator, with websites being lost due to hosting providers deleting the sites and databases. 

I first looked at an off the shelf website creator such as Square Space. I really like Square Space and feel that the online website design tools are effective in building sites quickly that look visually appealing. I had 3 major apprehensions: 

  • It does not look professional, or encourage people to download your app if you have a Square Space footer on your website. 
  • It would be harder to add more functionality later on driven from the API
  • It is quite expensive if the site doesn't take off

Ultimately for me, if this was about spinning up a business quickly, I would have overcome these apprehensions, and dealt with the consequences down the line. This however is a passion project, and therefore I have the time to do things right, and learn whilst I do them.

JAMSTACK

To the earlier point around administration, I wanted to avoid purchasing a physical server. It is expensive, difficult to manage and only has dis-benefits. I have been keen on a JAMSTACK website, as there are major cost, configuration and management benefits. Azure Files (and alternatives) are also good, but ultimately add complexity into the design without driving functionality.

Building from scratch a HTML/CSS/Javascript website that support JAMSTACK principles is relatively easy. However I was keen to look at a headless CMS to be able to easier manage content and ensure that others could update the website if needed at a later date.

There are many open source free options for headless CMS, with Umbraco being a clear leader for enterprise websites. I however decided on Publii which is unique in being a desktop CMS. As the website will only be updated (at the moment) by a small number of people, it is easier to install a desktop client and share the files across Dropbox/OneDrive than setup and administer an Umbraco installation. In the future, as the needs grow, it may be necessary to move to such a platform, but that won't be a problem for a long time.

Publii

Publii added more work on the development, as unless you are willing to use one of their templates (which aren't great!), you will need to code HTML/CSS/Javascript bottom up, and then the complexity of calling data from the CMS. I really like the way Publii works to bring the site and compile together. The major downside is the lack of concept of pages and is clearly built for blog posts. This is frustrating, but in consideration there are two types of pages - static webpages that convey information, and functionality webpages that do something. 

The posts function works well to turn the post into a static webpage. Functionality webpages are always going to need code added, so instead these pages were created separately within the template and configuration was added to the Theme to be able to tweak as needed. This is a fairly nice solution, but does add the problem that the Theme is largely unusable if you want to use it for a different project with different needs.
